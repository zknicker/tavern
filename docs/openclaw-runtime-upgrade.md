# OpenClaw Runtime Upgrade

Tavern Runtime owns the OpenClaw npm version it launches. Upgrades are Tavern migrations, not a
local operator task.

## Goals

- Keep `~/.tavern` backup/restore semantics intact.
- Preserve projected archives in `~/.tavern/tavern.sqlite`.
- Recreate or migrate managed OpenClaw runtime state under `~/.tavern/runtime/openclaw/run`.
- Keep the projection namespace stable as `tavern-openclaw-managed`.

## Upgrade Steps

1. Pick the target OpenClaw npm version.
2. Confirm the version exists:
   ```bash
   bun pm view openclaw versions --json
   ```
3. Update the root `openclaw` dev dependency to the exact version:
   ```bash
   bun add -d openclaw@<version>
   ```
4. Update `packages/tavern-openclaw-messenger` to the same peer/runtime version when needed, then
   run `bun install` so `bun.lock` records the workspace peer dependency.
5. Update `apps/runtime/src/openclaw/version.ts` and version-specific runtime tests when the
   fallback version or fixture paths contain the old version.
6. Read OpenClaw release notes for Gateway RPC, event, config, plugin, and state changes.
7. Update `apps/runtime/src/openclaw/config.ts` if generated config shape changed.
8. Update `packages/openclaw-gateway-adapter` mappings and tests for Gateway contract changes.
9. Update the website e2e OpenClaw mock provider if the pinned OpenClaw QA provider changed. The
   source is `extensions/qa-lab/src/providers/mock-openai/server.ts` from the matching OpenClaw
   Git tag. Preserve Tavern's local `./close-http-server.ts` import after refreshing it.
10. Add any needed one-time runtime-state migration under `apps/runtime/src/openclaw/`.
11. Run the managed runtime smoke with a copy of an existing `~/.tavern` directory when the upgrade
   touches state or config.
12. Run focused server/runtime typecheck, lint, unit tests, and live OpenClaw e2e coverage.

## State Rules

- `~/.tavern/runtime/openclaw/versions/<version>` is an npm cache and can be deleted/reinstalled.
- `~/.tavern/runtime/openclaw/run/openclaw.json` is generated by Runtime and can be overwritten.
- `~/.tavern/runtime/openclaw/run/state` is managed OpenClaw state. Migrate it only when OpenClaw
  requires durable state compatibility.
- `~/.tavern/tavern.sqlite` is Tavern durable state. Do not delete it during an OpenClaw upgrade.
- Projection rows remain scoped to `tavern-openclaw-managed` across upgrades.

## Migration Guidance

Prefer regenerating OpenClaw config over patching old config files. If OpenClaw changes a config
shape, update the generated config builder and let Runtime apply the new document.

Use runtime-state migrations only for OpenClaw-owned records that cannot be regenerated or resynced,
such as identity material, session indexes, or task databases. Migrations must be idempotent and
version-gated by observed state, not by a separate hand-edited flag.

Do not add server-side OpenClaw config fixups for new managed config behavior. The server should
read projections and expose Tavern-owned settings; Runtime should compose and apply OpenClaw config.

## Verification

Required checks for every OpenClaw version bump:

```bash
bun run --filter @tavern/runtime lint
bun run --filter @tavern/runtime typecheck
bun run --filter @tavern/runtime test
bun run --filter @tavern/server typecheck
bun run --filter @tavern/openclaw-gateway-adapter test
bun run test:e2e:live-openclaw
```

Run these commands from the repo root. Some filtered package commands execute with the package as
their working directory, but the shipped OpenClaw package is installed at the repo root.

For state/config upgrades, also smoke with a copied `~/.tavern` tree:

```bash
TAVERN_RUNTIME_ROOT=/tmp/tavern-upgrade-smoke/runtime \
DATABASE_PATH=/tmp/tavern-upgrade-smoke/tavern.sqlite \
bun run dev
```

Confirm that:

- Runtime status reports `agentRuntimeId: tavern-openclaw-managed`.
- Gateway starts through Seatbelt.
- Generated config contains Tavern Messenger and memory-core.
- Existing projected chats/sessions remain visible from the copied SQLite database.
- Current OpenClaw agents/chats/cron sync without deleting archived session/message history.
